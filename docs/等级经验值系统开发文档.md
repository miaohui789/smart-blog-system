# 等级经验值系统开发文档（适配 Smart-Blog-System）

## 一、目标与现状

### 1. 目标

在现有 `user_level` 体系基础上，补齐“经验值获取 → 等级提升 → 前端展示 → 可追溯流水”的完整闭环，优先支撑以下场景：

- 学习行为激励（学习一次、完成抽查）
- 社区行为激励（发文、评论、关注、互动）
- 用户成长可视化（当前等级、当前经验、升级进度）

### 2. 项目现状（已具备）

- 数据库 `sys_user` 已有 `user_level` 字段，范围为 1-100。
- 后端已有等级工具类 `UserLevelUtils`，包含等级归一化逻辑。
- 前端已有等级展示组件 `UserLevelBadge`、`VipUsername`，并在用户卡片、个人主页、关注列表等页面展示等级。
- 目前缺少统一经验值字段、经验流水表、经验规则和幂等控制。

### 3. 本次方案边界

- 不改动现有 VIP 逻辑（`vip_level`、`vip_expire_time`）。
- 等级系统作为独立成长线，保持与 VIP 并行。
- 先做“规则固定 + 后台可查看”，后续再做“后台可配置”。

***

## 二、总体设计

### 1. 核心对象

- 等级：`user_level`，沿用现有字段（1-100）。
- 当前等级经验：当前等级内已积累经验值。
- 累计经验：用户历史总经验。
- 经验流水：每次发放或扣减的明细记录。

### 2. 设计原则

- 幂等优先：同一业务事件只计一次经验。
- 上限保护：每日上限、单行为上限、防刷限制。
- 事务一致：行为成功后再加经验，避免“加了经验但业务失败”。
- 可观测：流水可查、规则可查、等级变化可追踪。

### 3. 升级模型

采用“配置表驱动”，不写死公式：

- 每个等级配置一个 `need_exp`（升到下一级所需经验）。
- 用户经验累计时，循环判断是否满足升级条件。
- 升级后扣除对应经验，剩余经验进入新等级继续判断。

优势：

- 运维可按阶段调优（早期升级快，后期升级慢）。
- 前后端统一读取配置或聚合结果，避免公式不一致。

## 三、数据库设计

延续项目当前风格：逻辑关联，不强制外键。

### 1. 改造 `sys_user`

新增字段：

- `current_exp` int default 0 comment '当前等级经验'
- `total_exp` int default 0 comment '累计总经验'
- `exp_update_time` datetime null comment '经验最近更新时间'

说明：

- `user_level` 继续作为唯一等级来源。
- `current_exp` 用于进度条；`total_exp` 用于排行与运营统计。

### 2. 新增表 `user_exp_record`

用途：经验流水（审计 + 去重 + 回溯）。

建议字段：

- `id` bigint PK
- `user_id` bigint 用户ID
- `biz_type` varchar(50) 业务类型（STUDY\_RECORD、CHECK\_FINISH、ARTICLE\_PUBLISH 等）
- `biz_id` varchar(64) 业务唯一标识（题目ID/任务ID/文章ID/日期）
- `exp_change` int 经验变化（正负）
- `before_level` tinyint
- `after_level` tinyint
- `before_exp` int
- `after_exp` int
- `remark` varchar(255)
- `create_time` datetime

索引建议：

- `idx_user_time(user_id, create_time)`
- `idx_biz_type_id(biz_type, biz_id)`
- `uk_user_biz(user_id, biz_type, biz_id)`（幂等核心）

### 3. 新增表 `user_level_config`

用途：等级经验阈值配置。

建议字段：

- `id` bigint PK
- `level` tinyint unique 等级（1-100）
- `need_exp` int 升到下一等级所需经验
- `title` varchar(50) 等级称号（可选）
- `status` tinyint default 1
- `create_time` datetime
- `update_time` datetime

初始化建议：

- 1-10 级：每级 80
- 11-30 级：每级 120
- 31-60 级：每级 180
- 61-90 级：每级 260
- 91-100 级：每级 400

## 四、经验规则设计（首版）

### 1. 发放规则（建议）

- 学习一次（`recordStudy`）：+2，单日上限 20
- 完成抽查任务（`finishTask`）：+10，单日上限 50
- 抽查高分奖励（任务最终分 >= 90）：额外 +5，单日上限 20
- 发布文章（审核通过或发布成功）：+15，单日上限 30
- 发布评论（有效评论）：+2，单日上限 20
- 关注用户（首次关注成功）：+1，单日上限 10
- 每日首次登录签到：+3，每日 1 次

### 2. 防刷规则（必须）

- 同 `biz_type + biz_id + user_id` 只发一次（唯一索引兜底）。
- 行为必须成功后才触发经验写入。
- 对“可频繁触发”的行为配置日上限。
- 被删除/撤销内容不自动扣回，首版保持简单；后续可加扣减补偿任务。

### 3. 等级边界

- 最低等级 1，最高等级 100。
- 达到 100 级后仅累积 `total_exp`，`current_exp` 可继续累加或固定为 0，建议继续累加方便展示成长。

## 五、后端实现方案（Spring Boot + MyBatis Plus）

### 1. 新增模块结构

建议新增：

- `entity/UserExpRecord.java`
- `entity/UserLevelConfig.java`
- `mapper/UserExpRecordMapper.java`
- `mapper/UserLevelConfigMapper.java`
- `service/UserExpService.java`
- `service/impl/UserExpServiceImpl.java`

### 2. 服务核心接口

`UserExpService` 建议提供：

- `grantExp(Long userId, String bizType, String bizId, int exp, String remark)`
- `getUserExpSummary(Long userId)`（等级、当前经验、下一等级所需经验、进度）
- `listUserExpRecord(Long userId, PageParam pageParam)`

`grantExp` 内部流程：

1. 参数校验与规则校验（日上限）
2. 幂等校验（`uk_user_biz`）
3. 读取用户当前 `user_level/current_exp/total_exp`
4. 依据 `user_level_config` 计算升级
5. 更新 `sys_user`
6. 写入 `user_exp_record`

### 3. 触发接入点（按当前项目代码）

优先接在 Service 成功分支：

- 学习记录：`StudyQuestionServiceImpl.recordStudy`
- 抽查完成：`StudyCheckTaskServiceImpl.finishTask`
- 点赞收藏：`ArticleLikeServiceImpl.like`、`ArticleFavoriteServiceImpl.favorite`
- 关注：`UserFollowServiceImpl.follow`

可先在 Controller 成功分支接入，再逐步下沉：

- 发文：`ArticleController.create`
- 评论：`CommentController.create`
- 登录：`UserController.login` / `loginByEmail`

### 4. 对外接口（新增）

前台用户端：

- `GET /api/user/exp/summary` 获取经验概览
- `GET /api/user/exp/records` 获取经验流水分页

后台管理端：

- `GET /api/admin/exp/rules` 查看经验规则
- `PUT /api/admin/exp/rules` 调整经验规则（二期）
- `GET /api/admin/exp/records` 查询用户经验流水

## 八、测试与验收

### 1. 单元/集成测试重点

- 同一 `biz_type + biz_id` 重复请求只加一次。
- 升级跨级场景正确（一次加经验升多级）。
- 达到 100 级后行为稳定。
- 高并发下经验不丢失、不重复。

### 2. 联调验收清单

- 完成学习行为后，前端经验实时变化。
- 升级后 `user_level` 与徽章展示一致。
- 流水记录与用户经验总量可对账。
- 日上限生效且有明确返回提示。

### 3. 回归范围

- 用户登录与资料接口
- 学习模块（学习、抽查）
- 文章互动（点赞、收藏、评论）
- 关注功能与通知功能

***

## 九、风险与规避

- 规则改动导致历史体验波动：采用“配置生效时间 + 灰度用户”策略。
- 并发重复加经验：以数据库唯一索引作为最终防线。
- 前后端等级定义不一致：统一以后端 `summary` 返回为准，前端仅渲染。
- 旧数据兼容问题：对 `current_exp/total_exp` 做默认值回填，避免空值。
