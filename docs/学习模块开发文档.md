# 学习模块开发文档

## 一、模块定位

本次新增模块为“学习模块”，建议在前台顶部导航新增一级菜单 `学习`，进入后拆分为两个子模块：

1. `面试题学习`
2. `面试题抽查`

当前题库内容来源于项目根目录下的 [面试题大全md](E:/1111xiangmu/Smart-Blog-System/new/smart-blog-system/面试题大全md)。

模块目标不是单纯展示题目，而是围绕“题库沉淀 + 用户学习进度 + 抽查复盘”形成一套可持续扩展的学习闭环。

---

## 二、业务设计

### 1. 面试题学习

面试题学习子模块建议提供以下能力：

- 按分类浏览题目，例如 Java基础篇、框架篇、数据库篇
- 支持题目搜索、关键词过滤、按掌握度筛选
- 支持查看标准答案、答案摘要、关键字
- 支持标记学习状态：未开始、学习中、已掌握、待复习
- 支持收藏题目
- 支持记录个人学习笔记
- 支持统计个人学习次数、最近学习时间

### 2. 面试题抽查

面试题抽查子模块建议提供以下能力：

- 随机抽查：系统随机从题库中抽取题目
- 专项抽查：只抽某个分类下的题目
- 收藏抽查：只抽取已收藏题目
- 薄弱题抽查：优先抽最近“模糊”或“忘记”的题目
- 作答模式建议采用“先自答，再看标准答案，最后自评”
- 自评结果分为：记得、模糊、忘记
- 抽查完成后生成本次结果统计，更新用户长期掌握情况

说明：
因为当前面试题大多是主观问答题，不适合直接做客观判分，所以第一版推荐使用“自评式抽查”。后续如果你想接 AI，可以在抽查明细表上扩展 AI 打分字段。

## 三、数据库设计

数据库脚本文件：[`docs/database/study_module.sql`](E:/1111xiangmu/Smart-Blog-System/new/smart-blog-system/docs/database/study_module.sql)

本次设计 5 张核心表：

1. `study_category`
2. `study_question`
3. `study_user_progress`
4. `study_check_task`
5. `study_check_task_item`

### 1. 表关系说明

- `study_category` 1 对多 `study_question`
- `study_question` 1 对多 `study_user_progress`
- `study_check_task` 1 对多 `study_check_task_item`
- `study_question` 1 对多 `study_check_task_item`
- `sys_user` 1 对多 `study_user_progress`
- `sys_user` 1 对多 `study_check_task`

说明：
项目现有数据库整体上更偏向“逻辑关联”而不是“强外键约束”，所以本次表设计延续现有风格，不强加外键，避免影响线上迁移和历史数据兼容。

### 2. 各表职责

#### `study_category`

用于管理题库分类和层级结构。

推荐用法：

- 一级分类固定为 `面试题题库`
- 二级分类对应当前 Markdown 文件，例如 `Java基础篇`、`框架篇`
- 后续如果你想继续细分到三级章节，也可以继续扩展 `parent_id`

#### `study_question`

用于存储题库主数据，是整个学习模块的核心表。

核心字段说明：

- `category_id`：所属分类
- `question_no`：题号，方便和 Markdown 原编号对齐
- `title`：题目标题
- `standard_answer`：标准答案
- `answer_summary`：答案摘要，适合列表页快速展示
- `keywords`：关键字，便于搜索
- `difficulty`：题目难度
- `source_file_name`、`source_section`：保留题目来源，便于后续回溯原始资料

#### `study_user_progress`

用于保存用户长期学习进度，一条记录对应“一个用户 + 一道题”。

核心字段说明：

- `study_status`：当前学习状态
- `mastery_level`：掌握度
- `study_count`：累计学习次数
- `remember_count` / `fuzzy_count` / `forget_count`：抽查结果累计
- `last_check_result`：最近一次抽查结果
- `is_favorite`：是否收藏
- `note_content`：个人笔记

这个表既能支撑学习页面，也能给抽查页面提供“薄弱题优先抽取”的依据。

#### `study_check_task`

用于保存一次完整抽查任务的头信息。

核心字段说明：

- `check_mode`：随机抽查、专项抽查、收藏抽查、薄弱题抽查
- `category_id`：专项抽查时选中的分类
- `question_count`：本次抽查题量
- `completed_count`：完成题量
- `remember_count` / `fuzzy_count` / `forget_count`：本次结果汇总
- `duration_seconds`：本次耗时

#### `study_check_task_item`

用于保存抽查任务内的题目明细和用户答题结果。

核心字段说明：

- `user_answer`：用户输入的答案，可为空
- `self_result`：自评结果
- `score`：本题得分
- `review_note`：本题复盘备注

第一版推荐评分规则：

- `记得 = 100`
- `模糊 = 60`
- `忘记 = 0`

## 四、题库初始化方案

当前项目中已经有一批 Markdown 题库文件：

- `Java八股文200道-Java基础篇.md`
- `Java八股文200道-框架篇.md`
- `Java八股文200道-数据库篇.md`
- `Java八股文200道-微服务篇.md`
- `Java八股文200道-MQ和ES篇.md`
- `Java八股文200道-Linux服务器篇.md`
- `Java八股文200道-补充篇.md`

SQL 脚本已经先帮你初始化了分类数据，但题目正文暂时没有直接入库。推荐后续按下面规则做批量导入：

### Markdown 解析规则建议

1. 文件名作为二级分类来源
2. `##` 标题可作为章节标题，写入 `source_section`
3. `###` 标题作为题目标题，写入 `title`
4. `###` 下面紧跟的正文或代码块内容写入 `standard_answer`
5. 原始序号提取后写入 `question_no`

### 导入结果示例

- `Java八股文200道-Java基础篇.md`
  对应分类：`Java基础篇`
- `### 1、JDK、JRE、JVM区别`
  对应题目：`title = JDK、JRE、JVM区别`
- 下面代码块内容
  对应标准答案：`standard_answer`

说明：
如果你后面需要，我还可以继续帮你补一个“Markdown 题库一键导入脚本”。

## 六、后端接口规划

本项目后端是 Spring Boot + MyBatis Plus，建议学习模块也沿用当前项目风格，采用：

- `controller/web` 提供前台接口
- `controller/admin` 提供后台接口
- `entity`、`mapper`、`service`、`service/impl` 按现有结构扩展

### 1. 前台接口建议

#### 学习分类与题库

## 九、补充说明

### 1. 为什么只设计 5 张表

因为你当前需求还是第一版，核心是先把：

- 题库沉淀下来
- 用户学习行为记录下来
- 抽查结果记录下来

这三件事做好。

如果第一版一开始就把错题本、排行榜、学习计划、AI评分、导入日志全部拆成十几张表，后面开发成本会明显上升，而且很多表短期用不上。

### 2. 为什么抽查不直接做自动判分

因为你现在的题库主要是主观题，例如“JDK、JRE、JVM 区别”“说说 equals 和 == 的区别”，用户很难像选择题那样直接自动判对错。第一版先做自评式抽查更贴合面试场景，也更容易上线。

### 3. 后续可扩展方向

如果后面你要继续升级，可以在 `study_check_task_item` 上扩展这些字段：

- `ai_score`
- `ai_comment`
- `ai_compare_result`

这样就能逐步演进成 AI 辅助面试训练模块。
