# Smart Blog System 项目面试回答指南

> 适用场景：校招 / 社招 Java 后端、全栈开发、博客社区类项目介绍、面试问项目亮点与优化
>
> 使用建议：不要逐字背诵，优先记住“项目定位、架构、亮点、优化、个人贡献”这 5 个维度，再结合你真实参与度表达。

---

## 1. 项目一句话介绍

这是一个前后端分离的多端博客社区系统，包含用户 Web 端、管理员后台和微信小程序端，核心功能除了常规博客内容管理外，还扩展了 AI 对话、私信通知、VIP 会员体系、面试题学习与抽查模块，以及基于 Elasticsearch 的全站搜索能力。

---

## 2. 适合面试开场的 1 分钟版本

你可以这样说：

> 我做的是一个前后端分离的博客系统，后端基于 Spring Boot 2.7、Spring Security、MyBatis Plus、MySQL 和 Redis，前端是 Vue3 + Element Plus，同时还做了管理后台和微信小程序端。  
>  
> 和普通博客项目相比，这个项目不只是文章发布和浏览，我还做了几块比较有业务特色的能力：一块是 WebSocket 实时私信和通知，一块是 AI 对话和模型切换，一块是面试题学习模块，支持题库导入、学习进度跟踪和抽查复盘。最近又把文章和面试题搜索升级成了 Elasticsearch 的统一搜索。  
>  
> 在项目实现上，我比较关注可扩展性和性能，比如做了 Redis 缓存、缓存失效控制、浏览量防刷、启动时 Token 清理、Redis 不可用时服务降级，以及针对 2 核 2G 服务器做过一套低内存部署优化。

## 3. 适合面试展开的 3 分钟版本

你可以这样说：

> 这个项目整体是一个内容社区 + 学习工具平台。  
>  
> 前台用户端支持文章浏览、分类标签筛选、点赞收藏评论、用户主页、私信通知、文章发布、AI 对话、VIP 会员、深浅色主题切换；后台支持文章、分类、标签、评论、用户、角色权限、系统版本、VIP、AI 配置、学习题库的管理；另外还有微信小程序端，覆盖文章浏览、用户中心、消息通知、VIP 和 AI 聊天。  
>  
> 技术架构上，后端是 Spring Boot 2.7 + Spring Security + JWT 做认证授权，MyBatis Plus 做数据库访问，MySQL 做主存储，Redis 做缓存与部分状态管理，WebSocket 做实时消息推送，SSE 做 AI 流式输出，Swagger/Knife4j 做接口文档。前端是 Vue3、Pinia、Vue Router、Element Plus，构建工具是 Vite。  
>  
> 在业务设计上，我做了几个比较有亮点的模块：  
> 1. 私信与通知：通过 WebSocket 建立用户级连接，支持实时推送和单设备登录控制。  
> 2. AI 对话：支持多模型配置、模型切换、每日调用限制、SSE 流式返回。  
> 3. 面试题学习：把 Markdown 题库结构化导入数据库，支持分类学习、关键词搜索、学习状态、收藏、笔记、抽查历史。  
> 4. 全站搜索：把文章和面试题统一接入 Elasticsearch，前端可以在同一个搜索入口同时搜文章、面试题和用户。  
>  
> 性能和稳定性方面，我做了 Redis 缓存、缓存穿透和失效控制、文章浏览量防刷、低内存参数调优、应用启动后清空旧 Token 避免单设备登录状态错乱，以及 Redis 不可用时的降级处理。  
>  
> 如果面试官继续深挖，我通常会从权限设计、缓存设计、学习模块的数据建模、以及 ES 搜索接入这几个点继续展开。

## 5. 项目架构怎么讲

## 5.1 总体架构

可以按下面这个逻辑描述：

```text
用户端 Web / 管理后台 / 微信小程序
            ↓
        Spring Boot API
            ↓
安全认证：Spring Security + JWT
            ↓
业务层：文章 / 用户 / 消息 / AI / 学习模块 / VIP
            ↓
MyBatis Plus + MySQL
            ↓
Redis 缓存 / WebSocket 实时推送 / SSE 流式响应 / Elasticsearch 搜索
```

### 5.2 多端架构

这个项目有 3 个客户端：

- `frontend/user`：用户 Web 端
- `frontend/admin`：管理员后台
- `miniprogram`：微信小程序端

这能体现你项目不是单页面 Demo，而是一个完整的多端业务系统。

### 5.3 后端分层

后端代码结构比较标准：

- `controller`：按 `web` 和 `admin` 拆分接口
- `service` / `service.impl`：业务逻辑层
- `mapper`：数据访问层
- `entity`：实体对象
- `config`：系统配置
- `security`：JWT 和认证拦截
- `websocket`：实时消息服务
- `task`：定时清理任务

这类结构在面试里很好讲，因为很符合中后台标准工程实践。

## 8. 你可以重点讲的技术亮点

### 8.1 JWT + Spring Security 的无状态认证

项目采用 JWT 作为登录态，后端通过 Spring Security 统一认证鉴权。

可以这样说：

> 我没有用传统 Session，而是用 JWT 做无状态认证。这样前后端分离更自然，也更适合多端接入。Spring Security 负责统一拦截请求，JWT 负责识别当前用户身份。

### 8.2 WebSocket 实时私信和通知

这个点很有面试价值，因为不是所有博客项目都有实时能力。

你可以这样说：

> 我们把用户连接按 `platform_userId` 管理，Web 端和管理端都能区分。用户上线时如果同账号已有旧连接，就先把旧连接关闭，保证单设备在线逻辑的一致性。这样不仅能做实时私信，也能做被踢下线提醒。

### 8.3 AI 对话流式输出

这个点体现现代项目特色。

你可以这样说：

> AI 聊天这块我用的是后端调用模型接口，再通过 SSE 把 token 流式返回前端。这样用户不用等整个回答生成完就能看到逐字输出，交互体验会好很多。同时我做了多模型配置和用户模型偏好切换。

### 8.4 面试题学习模块

这个点非常适合和“博客”形成差异化。

你可以这样说：

> 我没有把面试题简单做成一个静态页面，而是抽象成了学习模块。题目有分类、状态、学习次数、收藏、笔记和抽查结果，这样它就不只是知识展示，而是一个带学习闭环的系统。

### 8.5 Elasticsearch 统一搜索

这个点非常适合作为你最近的技术升级亮点。

你可以这样说：

> 原来文章和题目搜索都是数据库 `like`，数据一大性能和排序能力都会下降。后来我把文章和面试题都接进 Elasticsearch，前端统一通过一个搜索入口查询，支持更好的全文检索和后续扩展。为了兼容线上环境，我还保留了数据库兜底搜索。

## 9. 优化手段怎么讲

面试里你不能只说“用了 Redis”，要说“为什么用、解决了什么问题”。

### 9.1 Redis 缓存 + 降级设计

项目里的 Redis 不只是简单缓存，而是带降级策略：

- 应用启动时会检测 Redis 是否可用
- Redis 不可用时，不会让系统整体挂掉，而是以“无缓存模式”运行

你可以这样说：

> 我在 RedisService 里做了可用性检测和统一封装。这样 Redis 挂掉时不会影响主业务，只是退化成直接查库，保证系统可用性优先。

### 9.2 文章缓存体系

文章相关有多层缓存：

- 文章详情缓存
- 热门文章缓存
- 推荐文章缓存
- 分类/标签文章缓存
- 归档缓存

同时写操作后会统一清理相关缓存。

你可以这样说：

> 我不是只缓存一个接口，而是把文章详情、列表、热门、推荐、归档这些读多写少的数据都做了分层缓存，然后在文章更新、点赞、收藏、评论后统一做缓存失效控制，尽量保证一致性和性能平衡。

### 9.3 浏览量防刷与原子更新

这是一个很实用的细节。

项目里做法是：

- 用 `文章ID + IP` 作为短时访问标记
- 1 分钟内同一 IP 不重复计入浏览量
- 浏览量通过 SQL 原子自增

你可以这样说：

> 我对文章浏览量做了防刷处理，避免用户短时间刷新把浏览量刷高。实现上是 Redis 做短时访问标记，数据库层用原子自增避免并发丢计数。

### 9.4 启动时清理 Token，解决多设备登录状态错乱

这个优化挺有工程味。

项目启动时会清除旧 token，避免旧实例残留状态干扰新实例。

你可以这样说：

> 我发现单设备登录逻辑在服务重启后容易被旧 token 干扰，所以在应用启动阶段做了 token 清理，避免状态错乱和用户被误踢下线。

### 9.5 低内存环境优化

项目明确针对 `2 核 2G` 环境做过参数优化：

- Tomcat 线程数和连接数限制
- 响应压缩
- Jackson 序列化优化
- 数据库连接池收敛
- Docker 容器内存限制
- JVM 启动参数压缩

你可以这样说：

> 这个项目不是只在本地开发环境跑，我还考虑了低配置服务器部署，所以做过一套偏实战的资源优化，比如压低 Tomcat 线程数、限制连接池、开启响应压缩、调整 JVM 堆内存参数。

### 9.6 定时清理历史数据

学习抽查历史支持定时清理，避免数据无限膨胀。

你可以这样说：

> 我在学习模块里加了定时清理任务，按配置周期清理历史抽查数据，避免长期积累后影响表规模和查询性能。

## 11. 面试官常见追问和参考回答

### Q1：这个项目最大的难点是什么？

参考回答：

> 最大难点不是单个 CRUD，而是多个模块叠加后，如何保持系统结构清晰。比如博客、私信、AI、学习题库、会员、搜索这些模块其实差异很大，所以我在设计上尽量把控制器、服务层、缓存策略和多端能力分清楚，避免后期耦合太重。

### Q2：为什么要用 Redis？

参考回答：

> Redis 主要有三类用途。第一类是读多写少接口缓存，比如文章详情、热门文章、推荐文章；第二类是状态型数据，比如未读消息数、浏览量短时防刷；第三类是一些模块级缓存，比如学习模块总览和分类树。这样能明显减轻数据库压力。

### Q3：为什么消息系统要用 WebSocket？

参考回答：

> 因为私信和通知属于强实时场景，如果前端频繁轮询，性能和体验都不好。WebSocket 建立长连接后，后端有消息可以主动推送，尤其适合聊天、通知、被踢下线提示这种场景。

### Q4：学习模块为什么有价值？

参考回答：

> 因为它把“内容展示”升级成了“学习闭环”。用户不只是看题，还能记录学习状态、收藏、记笔记、抽查和复盘，这种数据结构后续还能接 AI 打分和个性化推荐，扩展空间比静态题库大很多。

### Q5：为什么引入 Elasticsearch？

参考回答：

> 因为数据库 `like` 搜索在数据量大时性能和相关性都有限，尤其面试题和文章都是长文本，更适合全文检索。引入 ES 之后，文章和面试题可以统一搜索，后面还可以继续做高亮、分词优化和搜索建议。

### Q6：你做了哪些稳定性设计？

参考回答：

> 主要有三类：  
> 1. Redis 不可用时的降级处理，保证主流程不挂。  
> 2. 启动时清理旧 token，避免多端登录状态错乱。  
> 3. 写操作之后做缓存失效和事务提交后的延迟清理，减少旧数据回填。

### Q7：为什么选择 MyBatis Plus？

参考回答：

> 这个项目业务表比较多，但很多查询还是以标准增删改查为主。MyBatis Plus 能减少样板代码，提高开发效率，同时保留了比较好的 SQL 可控性，适合这种业务系统。

> 我做的是一个前后端分离的多端博客系统，包含用户 Web、管理员后台和微信小程序。技术栈上后端用的是 Spring Boot、Spring Security、JWT、MyBatis Plus、MySQL 和 Redis，前端是 Vue3 + Element Plus。  
>  
> 这个项目和普通博客不一样的地方在于，我把它做成了“内容社区 + 学习平台”的结合体。除了文章、评论、分类标签这些基础能力之外，还做了 WebSocket 私信通知、AI 对话、VIP 系统，以及面试题学习和抽查模块。  
>  
> 技术上我重点做了几块：一是 JWT 无状态认证和权限控制，二是 Redis 缓存和缓存失效设计，三是 WebSocket 实时消息，四是 SSE 流式 AI 返回，五是把文章和面试题统一接入 Elasticsearch 做全文搜索。  
>  
> 优化方面，我做过浏览量防刷、数据库原子更新、Redis 不可用降级、启动时 Token 清理、学习模块缓存和定时清理任务，还有面向 2 核 2G 服务器的一套低内存部署参数优化。  
>  
> 如果面试官愿意深入，我比较想展开讲搜索升级、学习模块的数据建模，以及实时消息这三块，因为这三块最能体现业务设计和工程实现能力。

## 17. 最后建议

如果你时间有限，优先准备这 4 个问题：

1. 你这个项目是做什么的？
2. 你的项目架构是什么？
3. 你做了哪些有技术含量的点？
4. 你做了哪些优化？

只要这四个问题答顺了，你这个项目在面试里就已经很有说服力了。
