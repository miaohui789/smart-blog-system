### 185、Redisson 看门狗机制

```
Redisson 的看门狗（Watch Dog）机制，是为了解决分布式锁在持有锁的线程因意外（如宕机、网络中断）未主动释放锁时，导致锁永久无法释放的问题。
核心逻辑：当使用 Redisson 获取分布式锁时，若未指定锁的过期时间，看门狗会自动生效 —— 它会启动一个后台线程，每隔一定时间（默认 30 秒的 1/3，即 10 秒）检查持有锁的线程是否仍在运行。如果线程还在正常工作，看门狗会自动延长锁的过期时间（重置为 30 秒），避免锁提前释放；若线程意外终止，看门狗也会停止工作，锁到期后自动释放，防止死锁。
```

### 187、数据库连接池原理

```
数据库连接池核心是 “预先创建、复用连接”：
启动时创建一批连接，存为 “空闲池”；
应用需连接时，直接从池里拿（而非新建），用完放回池（而非关闭）；
自动管理连接数量（不超过上限），回收超时闲置连接，确保高效复用、控制资源。

目的：减少频繁创建 / 关闭连接的开销，提升性能和稳定性。
```

### 188、常见设计模式分类（23 种）

```
23 种设计模式通常分为三大类，分类及具体模式如下：
1. 创建型模式（5 种）
聚焦对象创建机制，解耦对象创建与使用：
单例模式（Singleton）：确保类仅一个实例，提供全局访问点
工厂方法模式（Factory Method）：定义创建对象的接口，由子类决定实例化哪个类
抽象工厂模式（Abstract Factory）：创建一系列相关 / 依赖对象，无需指定具体类
建造者模式（Builder）：分步构建复杂对象，分离构建与表示
原型模式（Prototype）：通过复制现有对象生成新实例
2. 结构型模式（7 种）
处理类或对象的组合，优化结构复用：
适配器模式（Adapter）：将一个类的接口转换为客户端期望的另一种接口
装饰器模式（Decorator）：动态给对象添加额外功能，不改变其结构
代理模式（Proxy）：为对象提供代理，控制对原对象的访问
外观模式（Facade）：为子系统提供统一接口，简化外部使用
桥接模式（Bridge）：将抽象与实现分离，使二者可独立变化
组合模式（Composite）：将对象组合成树形结构，统一处理单个对象和组合对象
享元模式（Flyweight）：共享细粒度对象，减少内存消耗
3. 行为型模式（11 种）
关注对象间通信与职责分配：
策略模式（Strategy）：定义算法族，动态选择使用哪种算法
模板方法模式（Template Method）：父类定义流程骨架，子类实现具体步骤
观察者模式（Observer）：对象间一对多依赖，一个对象变化时通知所有依赖者
迭代器模式（Iterator）：提供遍历集合元素的方法，无需暴露内部结构
责任链模式（Chain of Responsibility）：将请求传递给多个处理器，直到被处理
命令模式（Command）：将请求封装为对象，支持参数化、队列化请求
备忘录模式（Memento）：捕获对象状态并保存，以便后续恢复
状态模式（State）：对象状态变化时，动态改变其行为（类状态驱动）
访问者模式（Visitor）：在不修改类的前提下，为类添加新操作
中介者模式（Mediator）：用中介者封装对象间交互，减少直接依赖
解释器模式（Interpreter）：定义语言语法规则，构建解释器解析语句
```

### 189、HashMap为什么用红黑树不用 B树?

```
红黑树的 二叉轻量结构、内存操作高效性 以及 适合小规模数据的平衡特性，使其比 B 树更适合作为 HashMap 中链表的优化方案。
```

### 190、HashMap的长度为什么是 2的 N次方?

```
HashMap 长度设为 2 的 N 次方，核心是两点：
高效计算位置：length-1 二进制为连续的 1，用 hash & (length-1) 替代取模（%），位运算更快。
减少哈希碰撞：充分利用哈希值低位，分布更均匀；扩容时元素迁移逻辑更简单（新位置要么不变，要么 + 旧长度）。
最终目的是提升操作效率和减少冲突。
```

### 192、Vue 实例的生命周期

```
- beforeCreate():无法访问 data,methods,dom 中的元素，因为配置对象还未创建;
- created()：data 和 methods 里面的数据都已经初始化好了,我们可以调用了;
- beforeMount()：表示挂载之前运行,那么挂载点肯定没有应用到数据；这个阶段表示模版已经在内存中编辑完成,但是尚未把模版渲染到页面中,DOM 中的内容还没被替换,只是写了一些模版字符串；
- mounted()：表示内存中的编辑好的模版,已经挂载到页面中,用户可以看到渲染到页面的数据啦：
- beforeUpdate()：当执行 beforeUpdate()的时候,页面中显示的数据还是旧的,但 data 的数据是新的,只是还没有渲染到页面中
- updated()：当执行 updated 的时候,data 和页面中的数据都是最新的.
- beforeDestory():当经过某种途径调用$destroy 方法后，立即执行 beforeDestroy，一般在这里做一些善后工作，例如清除计时器、清除非指令绑定的事件等等。
- destroyed():当执行到 destroyed()方法时，Vue 组件里的滤器、指令、方法都不可用了。

**注：8 个生命周期函数,除了 beforeUpdate 和 updated 是多次执行;其他的都是只执行一次;**
```

### 198、单点登录（SSO）实现

```
单点登录（SSO） 是一种身份验证方法，允许用户只需登录一次，就可以访问多个相关但独立的应用程序，而无需为每个应用再次进行身份验证。它简化了用户体验，并减少了记住多个密码的需要。

最常用的是基于 OAuth2.0/OpenID Connect 的 SSO（第三方登录场景）。
核心原理：OAuth2.0 是授权框架（而非直接的认证协议），结合 OpenID Connect（基于 OAuth2.0 的身份层）可实现 SSO，适合对接第三方平台（如微信、GitHub 登录）。

流程（以 Authorization Code 模式为例）：
子系统 A 请求用户授权 → 重定向到认证服务器（如微信登录页）。
用户授权 → 认证服务器返回授权码（Code） 给子系统 A。
子系统 A 用授权码向认证服务器换取访问令牌（Access Token）。
子系统 A 用访问令牌向认证服务器获取用户信息 → 验证通过后允许访问。
其他子系统通过相同流程，用同一认证服务器的令牌实现 SSO。
```

### 200、TCP/IP模型

```
核心 4 层（从下到上）
网络接口层：负责物理网络连接（如网线、Wi-Fi），处理硬件相关的传输细节。
网络层：核心是 IP 协议，负责跨网络路由、寻址，让数据找到目标设备。
传输层：主要有 TCP（可靠传输）和 UDP（高速传输），负责端到端的数据传输控制。
应用层：直接面向用户，如 HTTP、FTP、DNS 等协议，提供具体应用服务。
```
