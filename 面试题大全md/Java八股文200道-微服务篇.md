### 161、**什么是微服务**

```
微服务架构是一种架构模式或者说是一种架构风格，它提倡将单一应用程序划分为一组小的服务，每个服务运行在其独立的自己的进程中，服务之间相互协调、互相配合，为用户提供最终价值。服务之间采用轻量级的通信机制互相沟通（通常是基于HTTP的RESTful API）,每个服务都围绕着具体的业务进行构建，并且能够被独立的构建在生产环境、类生产环境等。另外，应避免统一的、集中式的服务管理机制，对具体的一个服务而言，应根据业务上下文，选择合适的语言、工具对其进行构建，可以有一个非常轻量级的集中式管理来协调这些服务，可以使用不同的语言来编写服务，也可以使用不同的数据存储。
```

### 163、**SpringBoot**和SpringCloud的区别？

```
SpringBoot专注于快速方便的开发单个个体微服务。
SpringCloud是关注全局的微服务协调整理治理框架，它将SpringBoot开发的一个个单体微服务整合并管理起来，为各个微服务之间提供，配置管理、服务发现、断路器、路由、微代理、事件总线、全局锁、决策竞选、分布式会话等等集成服务。
SpringBoot可以离开SpringCloud独立使用开发项目， 但是SpringCloud离不开SpringBoot ，属于依赖的关系。
SpringBoot专注于快速、方便的开发单个微服务个体，SpringCloud关注全局的服务治理框架。
```

### 165、注册中心的作用？

```
注册中心的核心作用是实现分布式系统中服务的注册与发现，解决服务间调用的地址管理问题。

核心价值：
服务注册：服务启动时自动将地址、端口等信息登记到注册中心，无需手动配置。
服务发现：调用方通过注册中心查询目标服务地址，无需硬编码服务位置。
健康监测：实时感知服务节点的在线状态，剔除故障节点避免调用失败。
负载均衡：提供服务节点列表，支撑客户端或服务端的负载分发策略。

典型应用场景：
微服务架构中，大量服务间动态调用，无需人工维护服务地址映射。
服务集群扩容、缩容时，自动同步节点信息，保障调用稳定性。
跨环境部署时，简化服务地址切换，适配不同环境的服务部署情况。
```



### 166、配置中心的作用？

```
配置中心的核心作用是集中管理分布式系统中的配置信息，实现配置统一管控与动态更新。

核心价值：
统一管理配置，避免分散在各服务节点导致的维护混乱。
支持配置动态下发，无需重启服务即可生效，提升运维效率。
隔离环境配置，清晰区分开发、测试、生产等环境的配置差异。
保障配置安全，通过权限控制、加密存储等方式保护敏感配置。

典型应用场景：
分布式微服务架构中，统一管理多个服务的连接信息、业务参数。
需频繁调整配置的场景，如限流阈值、日志级别、开关功能等。
多环境部署时，快速切换不同环境的配置，减少手动修改错误。
```



### 167、网关的作用？

```
网关的核心作用是作为分布式系统的统一入口，对请求进行路由、过滤、管控，隔离内外网并简化服务调用。

核心价值
路由转发：接收客户端请求，根据规则分发到对应后端服务，隐藏服务真实地址。
请求过滤：统一处理认证授权、限流熔断、日志记录、跨域等通用逻辑，减少服务重复开发。
协议转换：适配客户端与后端服务的不同协议（如 HTTP 转 RPC），降低跨协议调用复杂度。
流量管控：监控请求流量，实现限流、熔断、负载均衡，保障系统稳定性。

典型应用场景
微服务架构中，客户端通过网关统一访问多个分散服务，无需记住多个服务地址。
需统一安全管控的场景，如接口认证、权限校验、敏感信息过滤。
跨终端、跨协议调用场景，如移动端 HTTP 请求调用后端 RPC 服务。
```



### 168、什么是服务雪崩效应？如何解决？

```
服务雪崩效应是某个核心服务故障后，故障沿依赖链扩散，导致整个分布式系统连锁崩溃的现象。

核心成因
服务间强依赖，无隔离机制，一个服务故障直接影响依赖它的上下游。
故障服务未及时熔断，大量请求持续涌入，耗尽服务器资源（线程、连接池等）。
缺乏限流措施，突发流量或故障后的重试请求加剧系统负载。

解决方案
熔断降级：通过熔断器（如 Sentinel、Hystrix）监控服务状态，故障时快速 “断开” 调用，返回降级响应（如默认值、缓存数据），避免故障扩散。
限流限流：限制单位时间内的请求量（如 QPS 阈值），防止突发流量压垮服务，常用令牌桶、漏桶算法。
服务隔离：采用线程池隔离或信号量隔离，为不同服务分配独立资源，避免单个服务故障占用所有资源。
超时重试控制：为服务调用设置合理超时时间，避免无限等待；重试需加退避策略（如指数退避），防止重复请求放大故障。
缓存与降级兜底：核心接口提前缓存结果，故障时直接返回缓存；非核心功能降级关闭，优先保障核心流程可用。
依赖治理：梳理依赖链，减少强依赖，核心服务采用集群部署提高可用性。

常用技术：Hystrix、Sentinel
```

### 170、分布式锁？

```
分布式锁是用于分布式环境下并发控制的一种机制，用于控制某个资源在同一时刻只能被一个应用所使用。
项目中，我一般使用Ridession实现的分布式锁。

Ridession原理：
Redis 的 SET 命令有个 NX 参数可以实现「key不存在才插入」，所以可以用它来实现分布式锁。
如果 key 不存在，则显示插入成功，可以用来表示加锁成功；
如果 key 存在，则会显示插入失败，可以用来表示加锁失败。
```



### 171、分布式事务？

```
分布式事务是解决分布式系统中多个服务跨数据源操作时，数据最终一致的机制，Seata 是主流的分布式事务框架，提供多种灵活模式适配不同场景。

Seata 的核心模式（4 种主流模式）
1. AT 模式（Auto Transaction）
核心逻辑：基于本地事务 + undo 日志 + 全局锁，无侵入业务代码。
执行流程：服务执行本地 SQL 并提交，记录 undo 日志；全局事务提交时删除 undo 日志，回滚时通过 undo 日志恢复数据。
适用场景：支持大部分关系型数据库（MySQL、Oracle 等），无业务代码侵入，适合微服务快速落地。
2. TCC 模式（Try-Confirm-Cancel）
核心逻辑：基于业务代码侵入式设计，分三阶段实现事务。
执行流程：Try 阶段预留资源（如冻结库存），Confirm 阶段确认操作（扣减库存），Cancel 阶段回滚资源（解冻库存）。
适用场景：非关系型数据库或无 undo 日志支持的场景，需开发者手动实现三阶段逻辑，灵活性高。
3. SAGA 模式
核心逻辑：基于长事务拆分，按业务流程顺序执行子事务，失败时通过补偿事务回滚。
执行流程：拆分分布式事务为多个本地子事务，每个子事务对应一个补偿事务；正常时按顺序执行子事务，异常时反向执行补偿事务。
适用场景：长事务场景（如订单履约、跨系统流程），允许短期数据不一致，追求最终一致。
4. XA 模式
核心逻辑：基于数据库原生 XA 协议（两阶段提交），依赖数据库自身事务支持。
执行流程：第一阶段（Prepare）所有数据库预提交事务，第二阶段（Commit/Rollback）根据预提交结果统一提交或回滚。
适用场景：需要强一致性的场景，性能较低，依赖数据库 XA 协议支持（如 MySQL InnoDB）。

Seata 模式选型建议
优先选 AT 模式：无侵入、开发成本低，适配大部分微服务场景。
强一致 + 非关系型数据库：选 TCC 模式，需自定义业务补偿逻辑。
长事务 + 最终一致：选 SAGA 模式，适合跨多个服务的复杂流程。
数据库原生支持 + 强一致要求：选 XA 模式，牺牲性能换一致性。
```



### 172、Jenkins自动部署原理

```
1. 配置 Jenkins 项目：在 Jenkins 上创建一个新项目，并设置相关参数，例如项目名称、源码管理方式、构建触发条件等。

2. 源码拉取：当触发条件满足时，Jenkins 会从Git仓库中获取最新的代码。

3. 构建过程：在构建过程中，Jenkins 将执行一系列操作来构建 Java 项目，例如：
¢设置构建环境：确定使用的 Java 开发工具、编译器版本等。
¢安装依赖：如果项目需要依赖其他库或框架，Jenkins 会通过构建工具Maven 下载和安装所需的依赖项。
¢编译项目：执行编译命令，将 Java 代码编译成可执行的字节码文件。
¢运行测试：执行单元测试、集成测试等，以确保代码质量和功能正确性。
¢代码打包：将编译后的代码打包成 JAR 文件，以备部署使用。

4. 部署到服务器：完成构建后，Jenkins 可以使用不同的方式将构建结果部署到服务器上，例如：
¢ 远程拷贝：通过 SSH 或其他协议将构建生成的文件复制到目标服务器上。
¢ 容器化部署：使用 Docker 或其他容器技术，在目标服务器上创建并运行包含应用程序的容器。

5. 验证部署：Jenkins 可以通过发送请求或执行自动化测试等方式验证部署的应用程序是否正常运行。

Jenkins 通过提供一个可定制的、持续集成和交付的自动化流程，结合不同的插件和工具，实现了 Java 项目的自动化部署。它可以自动化执行构建、测试和部署等任务，减少人工操作，提高交付速度和质量。同时，Jenkins 还支持日志记录、通知、报告生成等功能，方便开发团队进行监控和跟踪。
```





### 173、什么是链路追踪？

```
链路追踪的核心作用是监控分布式系统中请求的流转路径，记录各服务的调用关系、耗时及异常信息，快速定位问题根源。

核心价值
链路可视化：清晰展示请求从客户端到各个服务的完整流转路径，直观呈现服务依赖关系。
性能排查：统计每个服务节点的调用耗时，定位慢查询、卡顿等性能瓶颈。
问题定位：当请求失败时，追溯异常发生的服务节点、错误信息，缩短排查时间。
依赖分析：梳理服务间的调用频次、依赖强度，为架构优化提供数据支撑。

核心原理
生成全局追踪 ID：请求进入系统时分配唯一 ID，贯穿整个调用链路。
记录跨度（Span）信息：每个服务调用作为一个 Span，包含调用方、被调用方、耗时、状态等数据。
数据上报与存储：各服务将 Span 数据上报至追踪系统，统一存储和分析。
可视化展示：通过 UI 界面呈现链路拓扑、耗时分布、异常告警等。

主流工具
Zipkin：轻量易用，支持多种编程语言，适合中小型分布式系统。
SkyWalking：不仅支持链路追踪，还集成 APM（应用性能监控）、日志分析等功能。
```

### 175、什么是单例模式？几种写法

```
单例模式是保证一个类在整个系统中仅有一个实例，并提供全局唯一访问点的设计模式。

写法
饿汉式：类加载时直接创建实例，依赖 JVM 类加载机制保证线程安全。
懒汉式：延迟加载，首次调用 getInstance() 时才创建实例。线程不安全。
```

### 178、分支管理规范

![image-20251110183929739](Java八股文200道-微服务篇.assets/image-20251110183929739.png)

```
【master】主干永远是最新的代码，跟生产运行的代码保持一致，项目第一版一般只有master分支
【开发分支】每次开发新版功能时，都基于master创建新的分支：分支命名规则与业务功能有关，团队所有成员基于此分支开发
【测试分支】每次迭代开发完毕，完成自测和前后端联调后，基于开发分支创建测试分支
【修复分支】测试团队基于测试分支部署并进行测试，测试过程中有问题，继续修改开发分支，开发完毕再合并到测试分支
【生产分支】测试通过后基于测试分支新建生产分支(注意版本号，此分支代码方便回滚)， 基于生产分支进行部署，如果部署失败则回滚。如果部署成功，则将生产分支代码合并到master分支。
```
