### 51、SQL 的分类?

```
SQL（结构化查询语言，Structured Query Language）可以根据不同的功能和用途分为多个类型，主要包括以下几种：

1. 数据定义语言（DDL - Data Definition Language）
DDL用于定义和管理数据库结构，包括表、视图、索引、同义词、聚簇等数据库对象的创建、修改和删除。常见的DDL命令有CREATE（创建）、ALTER（修改）和DROP（删除）等。DDL语句主要用于定义数据库的整体结构和模式。
2. 数据操纵语言（DML - Data Manipulation Language）
DML用于对数据库中的数据进行操作，包括插入、更新、删除和查询数据记录。常见的DML命令有INSERT（插入）、UPDATE（更新）、DELETE（删除）。DML语句通常需要手动提交，即进行事务控制。
3. 数据查询语言（DQL - Data Query Language）
DQL主要用于从数据库中检索数据。它允许用户根据特定的条件来查询和获取保存在数据库中的数据。DQL的基本结构通常包括SELECT子句、FROM子句和WHERE子句，例如：“SELECT<字段名表>FROM<表或视图名>WHERE<查询条件>”。
4. 数据控制语言（DCL - Data Control Language）
DCL用于定义数据库用户的权限和控制数据库的访问。它允许数据库管理员授予和撤销用户的权限，以确保数据库的安全性和完整性。常见的DCL命令有GRANT（授权）和REVOKE（撤销权限）等。
5. 事务控制语言（TCL - Transaction Control Language）
TCL用于管理数据库中的事务，包括开始事务、提交事务和回滚事务等。常见的事务控制命令有BEGIN（开始事务）、COMMIT（提交事务）和ROLLBACK（回滚事务）。TCL确保数据库在事务处理过程中的一致性和完整性。
```

### 52、CHAR 和 VARCHAR有什么区别？

```
CHAR和VARCHAR是数据库中用于存储字符串数据的两种不同类型，它们之间存在几个关键的区别：

一、长度和存储方式
CHAR：
是固定长度的字符类型。
如果存储的字符串长度小于M，则剩余的空间会用空格填充。
VARCHAR：
是可变长度的字符类型。
VARCHAR(M)类型的字段会根据实际存储的字符串长度来分配空间。
二、使用场景
CHAR：
适用于存储长度固定且长度较短的数据，如身份证号码、电话号码等。
VARCHAR：
适用于存储长度可变的数据，如用户姓名、地址等。
```

### 53、MySQL 常用约束有哪些?

```
MySQL中常用的约束有以下几种：

1、主键约束（Primary Key Constraint）：
用于唯一标识表中每行记录。
主键列的值必须唯一且不能为NULL。
一个表只能有一个主键。
2、唯一约束（Unique Constraint）：
确保列中的值各不相同。
唯一约束允许空值（NULL），但空值不被视为重复。
一个表可以有多个唯一约束。
3、非空约束（Not Null Constraint）：
确保列中的值不能为NULL。
非空约束只能出现在表对象的列上。
4、外键约束（Foreign Key Constraint）：
用于关联两个表的列。
外键约束要求参照表中必须存在与被参照表中主键列相符的值。
外键约束用于确保数据表之间的数据完整性。
5、默认约束（Default Constraint）：
为列指定默认值。
当插入新行且未指定该列的值时，将使用默认值。
6、检查约束（Check Constraint）（注意：在某些MySQL版本中可能不支持或存在限制）：
确保列中的值满足某个条件。
检查约束可用于对表中数据进行更全面、更安全的控制。
7、自增约束（AUTO_INCREMENT）：
约束字段值自动递增。
通常与主键约束一起使用。
```

### 54、MySQL 索引的分类?

```
MySQL索引的分类确实可以从多个角度进行详细说明，以下是根据不同分类标准对MySQL索引的重新阐述：

一、按功能逻辑分类
普通索引：最基本的索引类型，主要任务是加快对数据的访问速度，没有任何限制。它可以在任何数据类型上创建，且允许索引列的值重复和为空。
唯一索引：确保索引列中的所有值均唯一，从而提高数据完整性。一张表中可以有多个唯一索引，但索引列的值必须唯一，允许有空值（但空值不被视为重复）。
主键索引：一种特殊的唯一索引，用于唯一标识表中的每一行数据。主键索引的列值不仅必须唯一，而且不能为空。在创建表时，通常会自动创建主键索引（如果指定了主键）。
全文索引：用于搜索文本数据，支持对文本字段的快速模糊搜索。它将文本分解成词语或短语，并对每个词语创建索引。全文索引通常用于CHAR、VARCHAR和TEXT类型的列。

二、按物理实现方式分类
聚簇索引（主键索引）：索引的叶子节点存储的是实际的数据行。在MySQL中，InnoDB存储引擎的聚簇索引是数据实际存储的顺序，因此主键索引（通常是聚簇索引）的查询效率非常高。
非聚簇索引（二级索引或辅助索引）：索引的叶子节点存储的是主键值或其他索引键的值，而不是实际的数据行。当使用非聚簇索引查询时，需要先找到对应的叶子节点，然后根据主键值再去聚簇索引中查找实际的数据行（这个过程称为“回表”）。

三、按作用字段个数分类
单列索引：建立在单个字段上的索引。这是最常见的索引类型，适用于查询中经常涉及单个字段的等值、范围或前缀搜索。
联合索引（复合索引）：建立在多个字段上的索引。
```

### 55、**MySQL 索引底层原理?**Mysql索引为什么用B+树？

```
MySQL索引底层原理
索引本质上是一种排好序的数据结构，用于快速定位表中的数据，避免全表扫描。它就像书籍的目录，通过目录可以快速找到目标章节，而无需逐页翻阅

MySQL选择B+树作为其索引数据结构，主要是基于以下几个原因：
1、性能高效：
B+树的非叶子节点不存储数据，只存储键值和子节点指针，这使得树的每一层能够存储更多的索引数量。
在相同数量的数据下，B+树的高度可能会比B树低，从而减少了磁盘I/O操作次数，提高了查询速度。
2、范围查询支持：
B+树的叶子节点通过双向链表相连，支持范围查询。这意味着只需要找到第一个符合范围条件的关键字，就可以通过指针一次性找到所有符合条件的关键字。
3、全局扫描效率高：
B+树遍历整棵树只需要遍历所有的叶子节点即可，不需要像B树一样需要对每一层进行遍历。
```

### 56、索引会失效的情况有哪些?

```
常见的索引失效情况：
（1）数据分布不均匀：
如果某个列的数据分布不均匀，例如某个值出现的频率过高或过低，索引可能无法有效地过滤掉大部分的数据，导致索引失效。
（2）使用了函数或表达式：
在查询中，如果对索引列使用了函数或表达式，如 SUBSTR(column, 1, 2) = 'ab'，索引将不会被使用，因为数据库需要计算每一行的值来确定是否满足条件。
（3）隐式类型转换：
当查询条件中的数据类型与索引列的数据类型不匹配时，数据库可能会进行隐式类型转换，这通常会导致索引失效。例如，如果索引列是字符串类型，而查询条件中使用了整数类型，就会发生隐式类型转换。
（4）使用了不等于操作符：
使用 !=、<> 或 NOT IN 等不等于操作符进行查询时，索引可能会失效，因为这些操作符需要扫描表中的大部分数据来确定是否满足条件。
（5）IS NULL 或 IS NOT NULL：
对索引列使用 IS NULL 或 IS NOT NULL 条件时，索引可能会失效，特别是在索引列包含大量NULL值的情况下。
（6）LIKE 通配符的错误使用：
当使用 LIKE 操作符进行模糊匹配时，如果匹配模式以通配符 % 开始，如 LIKE '%abc'，则索引会失效，因为数据库需要扫描整个索引来查找匹配的行。
（7）多表连接查询：
在多表连接查询中，如果连接条件没有正确使用索引，或者连接表之间的数据分布不均匀，索引可能会失效。
（8）复合索引顺序不正确：
创建复合索引时，如果索引列的顺序不正确，或者查询条件没有按照索引列的顺序进行匹配，索引可能无法被使用。
（9）索引列上有大量重复值：
如果索引列上有大量重复值，索引的过滤效果可能会降低，导致索引失效。
（10）查询条件不符合索引规则：
查询条件中没有使用到索引，或者使用了不符合索引规则的操作符，也会导致索引失效。
（11）索引过多或过少：
索引的数量过多或过少都可能导致索引失效。过多的索引会增加数据插入、删除和更新的开销，而过少的索引则可能无法充分利用索引的优势。
（12）索引碎片过多：
当索引碎片过多时，数据库可能会选择不使用索引，因为碎片化的索引可能导致查询效率降低。
```

### 57、SQL查询语句 优化方案?

```
常见的SQL查询语句优化方案：

一、索引优化
在WHERE及ORDER BY涉及的列上建立索引，可以显著提高查询速度。但应注意，索引并非越多越好，过多的索引可能会降低INSERT及UPDATE的效率。因此，应根据查询需求有针对性地创建索引。
二、查询语句优化
避免全表扫描：确保查询能够利用索引，避免对整张表进行扫描。可以通过EXPLAIN命令查看查询是否使用了索引。
选择适当的查询条件：
避免在WHERE子句中对字段进行NULL值判断，可以使用NOT NULL或特殊值（如0、-1）作为默认值。
避免使用!=或<>操作符，因为这些操作符通常不会使用索引。
慎用IN和NOT IN，对于连续的数值，可以使用BETWEEN代替IN。
避免在WHERE子句中使用OR连接条件，可以考虑使用UNION合并查询。
避免在WHERE子句中对字段进行函数操作或表达式计算，这会导致无法使用索引。
优化SELECT语句：
只选择需要的列，而不是所有列，可以减少网络传输的数据量，提高查询速度。
使用表的别名（Alias），可以减少解析的时间并减少由Column歧义引起的语法错误。
当只需要返回结果集的一部分时，可以使用SELECT TOP或LIMIT子句来限制返回的行数。
使用连接（JOIN）代替子查询：在某些情况下，使用连接代替子查询可以减少查询的复杂度，从而提高性能。
使用EXISTS代替IN：在某些查询中，使用EXISTS关键字代替IN关键字可以提高查询性能，因为EXISTS一旦找到匹配的行就会停止搜索，而IN则需要检查所有的匹配行。
优化GROUP BY语句：可以通过将不需要的记录在GROUP BY之前过滤掉来提高效率。
三、数据库配置和硬件优化
调整内存设置：确保有足够的内存来缓存数据和索引，这样可以减少磁盘I/O操作，提高性能。
优化磁盘I/O：使用快速的磁盘、优化磁盘布局（例如，将数据和日志文件放在不同的磁盘上）等可以减少磁盘I/O操作。
使用并行查询：对于大型查询，启用并行查询可以让多个处理器同时处理查询，从而提高性能。
```

### 58、什么是事务？事务的特性？

```
一、事务的定义
事务是程序中一系列严密的操作，这些操作被当作一个单独的逻辑工作单元执行。事务中的所有操作必须全部成功完成，否则在每个操作所做的更改将会被撤销。事务的概念在数据库管理中尤为重要，它确保了一系列数据库操作的原子性、一致性、隔离性和持久性。

二、事务的特性
事务具有四个特性，这四个特性通常被称为ACID特性：

原子性（Atomicity）：
事务是一个原子操作单元，其对数据的修改要么全部执行，要么全不执行。
如果一个事务在执行过程中出错，那么该事务会被回滚，即撤销它的所有操作。
一致性（Consistency）：
事务必须保证数据库的一致性。
在事务开始之前和事务结束以后，数据库的完整性没有被破坏。
这表示写入的任何数据都必须满足所有设置的规则，包括数据的完整性和约束性。
隔离性（Isolation）：
多个事务并发执行时，一个事务的执行不应影响其他事务。
隔离性是指并发中的事务，它们的执行结果应该与这些事务的执行顺序相同。
不同的事务拥有各自的数据空间，各自的操作不会对彼此产生影响。
持久性（Durability）：
一旦事务提交，则其结果永久保存在数据库中。
即使系统崩溃，重新启动后数据库还能恢复到提交事务后的状态，它的结果不会丢失。
```

### 59、mysql中的一些基本函数，你知道哪些？

```
MySQL中包含了丰富的常用函数和系统函数：

一、聚合函数
聚合函数主要用于对一组值执行计算，并返回单个值。
COUNT()：统计满足条件的行数。
示例：SELECT COUNT(*) FROM table_name; 统计表中的所有行数。
SUM()：计算数值列的总和。
示例：SELECT SUM(column_name) FROM table_name; 计算指定列的总和。
AVG()：计算数值列的平均值。
示例：SELECT AVG(column_name) FROM table_name; 计算指定列的平均值。
MAX()：找出数值列的最大值。
示例：SELECT MAX(column_name) FROM table_name; 找出指定列的最大值。
MIN()：找出数值列的最小值。
示例：SELECT MIN(column_name) FROM table_name; 找出指定列的最小值。

二、时间函数
时间函数用于处理日期和时间值。
CURRENT_DATE()：返回当前日期。
示例：SELECT CURRENT_DATE(); 返回当前日期。
NOW()：与CURRENT_TIMESTAMP()返回相同的格式。
示例：SELECT NOW(); 返回当前时间戳。

三、字符集函数（字符串函数）
字符集函数用于处理字符串数据。
CONCAT(str1, [,...])：连接多个字符串。
示例：SELECT CONCAT('Hello', ' ', 'World'); 返回Hello World。
LENGTH(string)：返回字符串的长度。
示例：SELECT LENGTH('Hello'); 返回5。
SUBSTRING(str, position[, length])：从str的position开始，取length个字符。
示例：SELECT SUBSTRING('Hello World', 7, 5); 返回World。
LTRIM(string)、RTRIM(string)、TRIM(string)：去除字符串前、后或前后的空格。
示例：SELECT LTRIM(' Hello'); 返回Hello。

四、数学函数
数学函数用于执行数值计算。
ABS(number)：返回number的绝对值。
示例：SELECT ABS(-10); 返回10。
RAND()：返回随机浮点数，范围在[0.0, 1.0]之间。
示例：SELECT RAND(); 返回一个随机浮点数。

五、其他函数
IFNULL(val1, val2)：如果val1为NULL，则返回val2，否则返回val1。
示例：SELECT IFNULL(NULL, 'Hello'); 返回Hello。
IF(test, v1, v2)：如果表达式test成立，返回结果v1；否则，返回结果v2。
示例：SELECT IF(1>0, 'yes', 'no'); 返回yes。
```

### 60、MySQL 怎么连表查询？

```
1. 内连接（INNER JOIN）
返回两个表中满足连接条件的匹配行。
sql示例：
SELECT columns
FROM table1
INNER JOIN table2 ON table1.common_column = table2.common_column;

2. 左连接（LEFT JOIN）
返回左表中的所有行，以及右表中满足连接条件的匹配行。右表中不匹配的行用NULL填充。
sql示例：
SELECT columns
FROM table1
LEFT JOIN table2 ON table1.common_column = table2.common_column;

3. 右连接（RIGHT JOIN）
与左连接相反，返回右表中的所有行，以及左表中满足连接条件的匹配行。左表中不匹配的行用NULL填充。
sql示例：
SELECT columns
FROM table1
RIGHT JOIN table2 ON table1.common_column = table2.common_column;

4. 全连接（FULL JOIN，MySQL不直接支持）
由于MySQL不直接支持FULL JOIN，可以通过UNION组合LEFT JOIN和RIGHT JOIN的结果来模拟。
sql示例：
SELECT columns
FROM table1
LEFT JOIN table2 ON table1.common_column = table2.common_column
UNION
SELECT columns
FROM table1
RIGHT JOIN table2 ON table1.common_column = table2.common_column
WHERE table1.some_column IS NULL OR table2.some_other_column IS NULL;
```



### 61、数据库三大范式是什么？

```
数据库三大范式简述：

第一范式（1NF）：要求数据库表中的每个字段都是原子的，不可再分。
第二范式（2NF）：在1NF基础上，要求表中的非主键字段完全依赖于主键，不能仅依赖于主键的一部分。
第三范式（3NF）：在2NF基础上，要求表中的非主键字段不传递依赖于主键，即非主键字段间不存在依赖关系。

这三大范式旨在减少数据冗余，提高数据一致性和查询效率。
```

### 62、MySQL 常用的存储引擎?InnoDB和MyISAM区别？

```
MySQL常用存储引擎简述：
InnoDB：默认引擎，支持事务，行级锁，高可靠性和性能。
MyISAM：适用于读多写少，表级锁，支持全文索引。
Memory：数据存内存，读写快，重启丢失，适合临时数据。
CSV：存为CSV文件，便于数据交换。
Archive：适用于大量历史数据，写入后不可改，高度压缩。
Federated：跨服务器访问数据，分布式存储。
NDB Cluster：高可用性、高扩展性分布式数据库系统。
Blackhole：丢弃所有数据，仅记录日志或测试用。
选择存储引擎需根据应用需求和性能要求。


InnoDB和MyISAM区别：
事务支持：InnoDB支持事务，MyISAM不支持。
外键支持：InnoDB支持外键，MyISAM不支持。
锁机制：InnoDB支持行级锁，MyISAM只支持表级锁。
索引类型：InnoDB是聚集索引，MyISAM是非聚集索引。
存储结构：InnoDB数据和索引存储在一起，MyISAM数据和索引分开存储。
性能：MyISAM读取性能通常较好，InnoDB写入和并发性能较好。
```

### 64、事务并发带来的问题有哪些？怎么解决？

```
MySQL事务并发问题：
1、脏读：事务读取另一事务未提交的数据，数据可能无效。
2、不可重复读：同事务两次读取同一数据，结果不一致。
3、幻读：事务读取数据时，另一事务插入新数据，导致结果集变化。

可以通过设置事务的隔离级别来消除并发问题，MySQL中常见的事务隔离级别有以下四种：
READ UNCOMMITTED（读未提交）
特性：允许读取未提交的数据。
并发问题：可能导致脏读、不可重复读和幻读。
应用场景：此隔离级别一般不推荐使用，因为它无法保证数据的一致性和完整性。
READ COMMITTED（读已提交）
特性：只能读取已经提交的数据。
并发问题：解决了脏读问题，但仍存在不可重复读和幻读问题。
应用场景：适用于对一致性要求不是特别高的场景，因为它可以提高并发性能。
REPEATABLE READ（可重复读）
特性：保证在同一个事务中多次读取同一数据时，数据是一致的。
实现机制：通过创建一致性视图和MVCC（多版本并发控制）机制来实现。
并发问题：解决了脏读和不可重复读问题，但仍存在幻读问题（在MySQL的InnoDB存储引擎中，通过next-key锁可以在一定程度上避免幻读，但在某些极端情况下仍可能发生）。
应用场景：MySQL的默认隔离级别，适用于大多数场景，因为它在保证数据一致性的同时，也提供了较好的并发性能。
SERIALIZABLE（可串行化）
特性：通过强制事务串行执行来避免并发问题。
并发问题：解决了脏读、不可重复读和幻读问题。
性能影响：由于事务需要串行执行，因此会大大降低并发性能。
应用场景：适用于对数据一致性要求极高，且对并发性能要求不高的场景。
```

### 66、讲一下binlog日志

```
Binlog是MySQL的二进制日志，记录所有修改数据库的操作（增、删、改等），以二进制格式存储。

主要功能：

```

### 67、MySQL主从复制了解吗

```
MySQL主从复制是一种数据同步机制，它允许将主数据库（Master）的数据更改同步到一个或多个从数据库（Slave）。

原理：
主从复制依赖于二进制日志（Binary Log）。主数据库在事务提交时，会将数据变更记录在二进制日志中。从数据库则通过I/O线程连接主数据库，请求并接收这些二进制日志事件，然后将这些事件记录在中继日志（Relay Log）中。接着，从数据库的SQL线程会读取中继日志中的事件，并在本地执行，以保持数据的一致性。
```

### 68、介绍MVCC实现原理

```
MVCC（多版本并发控制）原理简述：

MVCC通过维护数据的多个版本来实现并发控制。其核心组件包括：
版本链：每条记录都有多个版本，通过回滚指针连接成链表。新版本包含最新的事务ID，旧版本保存在Undo Log中。
Undo Log：记录数据的逻辑变化，用于回滚和版本链的构建。
Read View：事务快照读时生成的视图，包含活跃事务链表，用于判断记录的可见性。

MVCC的工作流程：
写操作：修改记录时，生成新版本并链接到版本链，同时记录Undo Log。
读操作：根据Read View判断记录的可见性，选择适当的版本读取。

MVCC的作用：
提升并发性能，允许读写操作并行执行。
解决脏读、幻读和不可重复读等事务隔离问题。
```

### 69、JDBC 操作数据库的步骤?

```
JDBC操作数据库的步骤可以概括为以下6步：

1、加载驱动：使用Class.forName()加载数据库驱动。
2、建立连接：通过DriverManager.getConnection()方法，使用数据库的URL、用户名和密码建立连接。

6、关闭资源：关闭ResultSet、Statement和Connection对象以释放资源。

```

### 70、Statement、PreparedStatement 的区别?

```
1、安全性：
Statement：易受SQL注入攻击。
PreparedStatement：通过参数化查询防止SQL注入。
2、预编译：
Statement：不预编译SQL，每次执行都需解析。
PreparedStatement：预编译SQL，提高执行效率。
3、参数化：
Statement：不支持参数化，需手动拼接SQL。
PreparedStatement：支持参数化，使用占位符设置参数。
4、执行效率：
Statement：执行效率较低，需重复解析SQL。
PreparedStatement：执行效率高，可重用预编译的SQL。
5、功能：
Statement：功能简单，适用于简单SQL。
PreparedStatement：功能强大，支持复杂SQL及批量操作。

简而言之，PreparedStatement在安全性、效率、功能和灵活性上均优于Statement，是JDBC编程中的首选。
```

### 75、Redis 的缓存雪崩、缓存穿透、缓存击穿?

```
1、缓存穿透是指查询一个不存在的数据（redis和mysql中都没有），就会导致每次请求都去查数据库，访问量大了可能压垮数据库。
解决：
（1）**缓存空数据**
查询返回的数据为空，仍把这个空结果进行缓存
（2）**布隆过滤器**
采用布隆过滤器，使用一个足够大的bitmap，用于存储可能访问的key，不存在的key直接被过滤；

2、缓存击穿是指缓存中没有但数据库中有的数据（一般是缓存时间到期），这时由于并发用户特别多，同
时读缓存没读到数据，这些请求都会击穿到DB，造成瞬时DB请求量大、压力骤增，这就是缓存被击穿。
解决方案：
（1）**设置热点数据永不过期**：内存占用率高。
（2）**互斥锁**：保证了数据的强一致性，但性能会差。

3、**缓存雪崩**是指在同一时段大量的缓存key同时失效或者Redis服务宕机，导致大量请求到达数据库，带来巨大压力。
解决方案：
（1）缓存数据的过期时间设置随机，防止同一时间大量数据过期现象发生。
（2）双缓存。
```

### 76、过期数据删除策略

```
（1）定时删除
到时间，直接删除。
创建一个定时器，到时间删除对应的键值对。
（2）惰性删除
每次访问某个键值对前，先判断这个键值对是否已经过期；
如果过期，删除，不能访问。
如果没有过期，可以访问。
**缺点**：如果一直没有访问这个键值对，这个键值对将一直占内存。
（3）定期删除
每隔一段时间，删除一次过期的数据。
```

### 77、内存淘汰策略

```
内存占用超过阈值，为了防止内存溢出，redis会将一部分数据清除。
（1）**针对设置了过期时间的键**
- `volatile-lru`：仅对设置了过期时间的键，淘汰最近最少使用的键。
- `volatile-ttl`：仅对设置了过期时间的键，淘汰剩余时间最短的键。
- `volatile-random`：仅对设置了过期时间的键，随机淘汰。
- `volatile-lfu`：仅对设置了过期时间的键，淘汰访问频率最低的键。
（2）**针对所有键**
- `allkeys-lru`：对所有键，淘汰最近最少使用的键。
- `allkeys-random`：对所有键，随机淘汰。
- `allkeys-lfu`：对所有键，淘汰访问频率最低的键。
（3）**无淘汰策略**
- `noeviction`：不进行淘汰，当内存超限时直接返回错误。默认。
```

### 78、什么是bigkey？会有什么影响？如何处理

```
1、什么是BigKey？
BigKey在Redis中通常指的是那些占用内存空间过大的键。这些键可能因为其值的数据量过大，或者键中包含的成员数量过多而被认为是BigKey。

2、BigKey对Redis系统的影响：
（1）网络阻塞：当对BigKey执行读请求时，由于需要传输大量的数据，可能会占用大量的网络带宽，导致Redis实例乃至所在物理机的性能下降。
（2）数据倾斜：BigKey所在的Redis实例内存使用率可能会远超其他实例，使得数据分片的内存资源无法达到均衡，破坏了集群的负载均衡。
（3）Redis阻塞：对于元素较多的hash、list、zset等数据结构，进行运算可能会耗费较长的时间，导致Redis主线程被阻塞。
（4）CPU压力：对BigKey的数据进行序列化和反序列化会导致CPU的使用率飙升，影响Redis实例和本机其他应用的性能。

3、处理BigKey的主要策略包括以下几个方面：
（1）拆分：将存储在大键中的数据拆分成多个小的值进行存储。例如，如果大值是一个大的JSON对象，可以通过使用MSET命令将该键的内容拆分存储到各个实例中；如果大值是一个大的列表，可以将其拆分为多个小的列表进行存储。
（2）修改数据结构：根据BigKey的数据类型和分析结果，考虑修改其数据结构以更适合存储大量数据。
（3）使用压缩算法：对于字符串类型的BigKey，可以考虑使用压缩算法（如Snappy、LZF等）对其进行压缩，以减少内存占用。但需要注意，压缩后的数据在读取时需要解压缩，可能会增加CPU负载。
（4）分片处理：如果BigKey的数据量非常大，可以考虑将其分片存储到多个Redis实例中，以降低单个实例的内存压力。
```

### 79、Redis 的应用场景?

```
1、数据缓存：
Redis 最常见的用途是作为缓存层，用于存储临时数据以减少对数据库的访问频率，提高系统的响应速度和吞吐量。
2、会话存储：
Redis 可以用于存储 Web 应用程序的会话数据，如用户登录信息、购物车内容等。由于 Redis 支持快速读写和持久化，因此非常适合这种需要快速访问和更新的场景。
3、分布式锁：
Redis 提供了基于 setnx 命令的分布式锁实现，可以确保在分布式环境中对共享资源的互斥访问。

4、发布/订阅模式：
Redis 的发布/订阅模式允许发送者（发布者）将消息发送到频道，而接收者（订阅者）可以订阅这些频道并接收消息。这种模式适用于实时消息传递、事件通知等场景。
5、计数器：
Redis 支持多种数据类型，包括字符串、哈希、列表、集合和有序集合。利用这些数据类型，Redis 可以轻松实现计数器功能，如网站访问量统计、用户点赞数统计等。
6、排行榜：
Redis 的有序集合（Sorted Set）数据结构可以非常方便地实现排行榜功能，如游戏排行榜、文章阅读量排行榜等。
7、地理空间索引：
Redis 支持地理空间索引（Geospatial Index），可以用于存储地理位置信息，并实现诸如附近地点搜索、计算两点之间的距离等功能。
8、消息队列：
虽然 Redis 不是专门的消息队列系统，但它可以利用列表（List）或流（Stream）数据结构实现简单的消息队列功能，用于任务调度、日志收集等场景。
```

### 80、如何保证redis和mysql的一致性？

```
1、延时双删策略：
在更新数据库之前先删除Redis中的缓存数据，更新数据库后再延时一段时间（通常等于读业务逻辑数据的耗时加上几百毫秒）再次删除Redis中的缓存数据。
优点：能够确保在并发读写场景下数据的一致性。
缺点：增加了删除缓存的操作次数，可能影响Redis的性能。

2、Canal组件：
使用Canal组件监控MySQL的binlog日志，将更新后的数据同步到Redis中。
Canal模仿了MySQL的slave数据库的备份请求，使得Redis的数据更新达到了与MySQL相同的效果。

```
